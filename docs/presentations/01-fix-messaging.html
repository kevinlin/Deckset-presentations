<h1>Characteristic of FIX</h1> <ul> <li>Simple, plain text, field-based text messages</li> <li>Platform and Transport independent, compatible with almost every common network technology</li> <li>Public standard<ul> <li>Owned &amp; managed by <a href="https://www.fixtrading.org/">FIX Protocol Ltd</a>, a non-profit financial community organization</li> </ul> </li> <li>Wide support and lots of software vendors availability</li> </ul>

<h1>FIX message structure</h1> <ul> <li>Header fields<ul> <li>message type, length, sequence number, sender/target, encoding, etc.</li> </ul> </li> <li>Body fields<ul> <li>session &amp; application data</li> </ul> </li> <li>Trailer fields<ul> <li>Signature, checksum</li> </ul> </li> </ul>

Slide 16

16 of 37
<h1>Message fields</h1> <ul> <li>For each field <code>&lt;Tag&gt;=&lt;Value&gt;&lt;Delimiter&gt;</code> the specification defines:<ul> <li>Tag – A unique number.</li> <li>Field Name – Field name with no spaces.</li> <li>Required - Mandatory/optional</li> <li>Description – Definition of data, data type, etc.</li> </ul> </li> <li><code>&lt;Delimiter&gt;</code> is ASCII SOH character</li> </ul>

<h1>FIX Versioning</h1> <p>FIX currently supports three versions of the application messages that serve as the basis for standardization:</p> <ol> <li>Version 5.0 Service Pack 2 <strong><em>FIX.5.0SP2</em></strong> is the most current version of FIX that supports multiple asset classes and a wide range of trading life cycle business processes. The FIX session protocol was separated from the application with the introduction of FIX.5.0</li> <li>Version 4.4 <strong><em>FIX.4.4</em></strong> is one of the most widely adopted versions oF FIX. FIX.4.4 covers multiple asset classes including fixed income instruments. As of 2010, it is considered compliant to use fields, messages, and components as of the latest extension pack of FIX with FIX.4.4.</li> <li>Version 4.2 <strong><em>FIX.4.2</em></strong> is widely used for equities, FX, and listed derivatives trading.</li> </ol>
